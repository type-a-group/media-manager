1.0 — The Reader Package

Read-only host integration · Item 33 v1: build-time / static · data layer only · monorepo subpath

Sub-project: docs/plans/v1.0/npx/ · Working agreement: docs/plans/v1.0/README.md · Phase 2 deliverable (review & approve before any code).

Goal (scoped in the Phase-1 interview). Let a host build (e.g. nicb.at) consume a media-manager workspace's on-disk format to render galleries and record lists without booting the editor — pure, read-only, and layout-aware so the host never hand-codes a path that rots when media-manager's on-disk format moves. v1 ships a data layer only (no components), for the build-time / static consumption mode, as a subpath export of this monorepo.
Why this is urgent, not speculative. nicb.at's readers (src/lib/index.ts: fetchImageList / fetchProjects / fetchQuotes) still import the pre-Item-18 paths — photos/image-data.json, files/*, projects/data.json, quotes/data.json — but the workspace on disk was migrated to the file-first layout (media/manifest.json + media/classes/photos.json + media/files/*, records/projects/data.json). Photos / Projects / Quotes are reading nothing right now. This is exactly the silent breakage the reader exists to prevent.

The framing correction (vs. the old Item 33 stub)

The pre-existing stub centred on a "root-threading refactor" — making paths.ts take an explicit root so a Node API can fs-read the workspace. Investigating nicb.at showed that's not how the target host consumes data. A static build reads JSON via Vite's import.meta.glob/import() and resolves image URLs through Vite's asset pipeline (hashed, content-addressed, bundled). There is no fs, no root path, no process.env at runtime. An fs-based openWorkspace(root) literally cannot emit the hashed asset URLs a static build needs.

So the old plan conflated three reader shapes:

ShapeHow it readsAssetsFits
(a) fs openWorkspace(root) Node fs reads the root Can't produce bundled URLs Static-export CLI (Item 7), a live server. Needs root-threading.
(b) Vite-bundle library v1 Host hands in import.meta.glob results; reader does pure-data joins Host's bundler hashes them; reader maps filename → url nicb.at & the reference repo. No fs, no root-threading.
(c) runtime HTTP client fetch a live /api/* Server streams [id]/blob A host running MM live in prod — deferred (Mode 2)

Our v1 decision (build-time / static, data-only, host resolves assets) is squarely (b). The root-threading refactor demotes to a later concern shared by (a)/Item 7 — not the heart of this.

Decisions locked (Phase 1)

AreaDecision

The hard problem: asset resolution

JSON is trivial; assets are the whole game. A class record gives you a file_id; the manifest maps it to file_name + intrinsic dims. But media/files/superpuff.jpg must become a hashed URL Vite serves — and only the host's own Vite instance can do that. import.meta.glob paths must be string-literal and relative to the calling module, so a library buried in node_modules cannot glob the host's asset tree. The reader API must be designed around the host passing in its own glob result.

nicb.at already wrestles with the secondary gotcha: filename extension case. Its filenameWithLowercaseExt() patch exists because a manifest file_name of olympic_river.jpeg must still match a glob key of olympic_river.JPEG. The reader bakes this tolerance in so every host doesn't reinvent it.

Architecture — a facade over a pure core

The public surface is one object with a handful of obvious methods. Everything below it — the manifest join, url/list normalization, ext-case asset matching, version guard — is private internals the developer never sees. Three layers, only the first is public:

1 · MediaManager facade the whole public API

Loaded once; exposes .media() / .records() / .globals() / .file() and returns ergonomic Collections of flat items. This is all a developer learns.

2 · Pure core private, env-agnostic

The transform functions (join, normalize, resolve, filter) — now internals of the facade. Reuse src/lib/core/ Zod DTOs + normalization so semantics match the editor exactly.

3 · Version guard

Read manifest.version (currently 2); reject an unsupported workspace loudly at load() — clear error, not a silent empty render.

Load once — the only glue the developer writes

import { MediaManager } from 'media-manager/reader/vite';

// two globs in; not async — eager globs are already resolved by Vite.
const mm = MediaManager.load({
  data:  import.meta.glob('$assets/media_manager/**/*.json', { eager: true, import: 'default' }),
  files: import.meta.glob('$assets/media_manager/media/files/*', { eager: true, query: '?url', import: 'default' }),
});

Two globs (not four): JSON must be parsed, images must be ?url'd — Vite needs those as separate calls, and the paths must be string-literal (Vite's hard constraint). The reader infers what each entry is — manifest vs. class vs. record vs. asset — from its path, so this snippet is identical for every host; only the workspace prefix changes.

Methods on the object

mm.media()              // every blob in the workspace        → Collection<MediaItem>
mm.media('photos')      // members of the "photos" class       → Collection<MediaItem>
mm.records('projects')  // records of a type                   → Collection<Record>
mm.globals()            // the globals singleton                → Record
mm.file(id)             // look up one blob by manifest id      → MediaItem | null
mm.classes()            // [{ id, name, icon, count }]
mm.types()              // [{ id, name, count }]

Flat item shapes — no plumbing leaks out

MediaItem {
  id, src, filename, width, height,   // src = resolved hashed URL, or null if the blob is missing
  classes: string[],                  // membership
  missing: boolean,
  fields: { … },                      // the class metadata, normalized
  field(key),                         // convenience getter (handles url / list values)
  file(key), files(key),              // follow a file-type field → MediaItem(s)
}
Record { id, lastModified, fields, field(key), file(key), files(key) }

Filtering — a tiny fluent Collection (five methods, not a DSL)

mm.media('photos')
  .where({ hidden: false })       // field equality (looks in fields first, then top-level)
  .where('Year', '2024')
  .sortBy('Year', 'desc')         // also .filter(fn) · .find(fn) · .first()
  // a Collection is iterable + has .length + .all — use it straight in {#each}

.map() returns a plain array (you're leaving for your own shape); .where / .filter / .sortBy return a Collection so they chain.

Following file references — never juggle an id

A file-type field stores a raw manifest id on disk. .field(key) returns that stored value; .file(key) resolves it the rest of the way:

const p = mm.records('projects').first();
p.field('thumbnail')   // → "8e6973f2-…"  (the stored id)
p.file('thumbnail')    // → MediaItem { id, src, width, height, … }  ← same shape as mm.media()
p.file('thumbnail')?.src
p.files('gallery')    // list-of-files field → Collection<MediaItem>

File references resolve to the same MediaItem objects as mm.media() (identity by id) — one consistent file shape whether reached via class membership, mm.file(id), or a file field. A dangling id yields null / missing: true, never a throw or a broken <img>.

Proof: nicb.at migration (the acceptance target)

v1 is "done" when nicb.at's three hand-rolled readers collapse onto the facade and render again — the ~40 lines of glob-juggling + ext-lowercasing + manifest-joining in src/lib/index.ts become:

// photos
const photos = mm.media('photos')
  .where({ hidden: false })
  .map((m) => ({ src: m.src, width: m.width, height: m.height,
                 alt: [m.field('name'), m.field('Location'), m.field('Year')].filter(Boolean).join(', ') }));

// projects
const projects = mm.records('projects').sortBy('date', 'desc');

// quotes
const quotes = mm.records('quotes').where({ show_on_homepage: true });

The host keeps its domain mapping + its own components (Gallery, ProjectCard) — that's the "data layer only" line. The reader stops at clean, typed, asset-resolved items.

Documentation — a first-class deliverable

The library is only as good as a developer's first 10 minutes with it. Documentation is part of the acceptance criteria, not an afterthought — a host developer (your friend, future-you) should go from zero to a rendered gallery without reading the source.
  • Reader README (src/lib/reader/README.md, shipped in the package): the copy-paste MediaManager.load() snippet, the full method table, item shapes, the five Collection methods, the file-reference story, and the version-guard error — each with a runnable example. Leads with a complete "photos gallery in 15 lines" walkthrough.
  • Exhaustive JSDoc on every public symbol (MediaManager, load, media/records/globals/file, Collection, MediaItem, Record) — matching this repo's rich-JSDoc convention (@param, use-case, "Concerns / future improvements"). This drives editor IntelliSense, which is the real docs surface.
  • Fully-typed surface — .d.ts for the subpath so hosts get autocomplete on mm., item fields, and Collection chains. Types are documentation here.
  • A worked recipe per shape in the README: a class gallery, a record list, following a file field, and handling missing assets — mirroring the nicb.at proof so the canonical example is real, not toy.
  • Migration note: "replacing a hand-rolled reader" — the before/after from nicb.at's index.ts, so the next host with stale readers has a map.

Verification gates

npm run check + npm run lint clean for touched files.
Purity: the reader core runs with no MEDIA_MANAGER_ROOT, performs no fs / network / writes (assert a fixture workspace is byte-identical before/after a full read sweep — the read-only guarantee).
Join correctness: a member missing from the manifest is flagged missing (not crash); a manifest entry with no class membership is omitted from a class view; url/list values normalize identically to the editor's read path.
Asset resolution: file_name foo.JPEG resolves against a glob key foo.jpeg and vice-versa; an asset with no glob entry yields src: null + missing: true, never a broken <img>.
Version guard: a non-v2 / unrecognized workspace throws a clear, actionable error (names the expected layout), not an empty result.
The nicb.at proof: with the reader wired in, Photos renders the photos class (hidden filtered, dims correct), Projects + Quotes render from records/* — from the current file-first data, no manual paths.
Documentation: reader README present with the load snippet + per-shape recipes + migration note; every public symbol has JSDoc; the subpath ships .d.ts so mm. autocompletes in a host editor. A dev can reach a rendered gallery from the README alone.
Docs in the same change: FEATURES.md reader entry + README section; Item 33 → Shipped & folded; FUTURE_CHANGES_TRIAGE.html synced.

Cut line (explicitly out of v1)

  • Components — read-only <Gallery>/lightbox/record cards/field renderers over the in-house Masonry.svelte. The natural v2 of the reader; the host renders for now.
  • Runtime HTTP-client mode (c) — for a host running MM live. Shares only core/ DTOs; addable later with zero rework to the v1 core. Capture as a follow-up backlog item.
  • fs openWorkspace(root) + the root-threading refactor — belongs to the static-export CLI (Item 7) and a live server, not the static-build face. Same pure core feeds it when built.
  • Responsive / multi-resolution assets (the Photos page's "load lower quality by default" TODO) — an asset-pipeline concern shared with Items 7 / 15; design once, later.
  • Registry publishing of the subpath (Item 43) — local-dep / file: link for now.
