1.0 — Making media-manager easier to run

Combined plan for the npx run trio · Items 30 · root discovery + verbs 31 · ephemeral port 32 · quiet heal

Sub-project: docs/plans/v1.0/npx/ · Working agreement: docs/plans/v1.0/README.md · Phase 2 deliverable (review & approve before any code).

Goal (scoped in the Phase-1 interview). One command, run from inside your repo, that finds your data, grabs a free port, opens when actually ready, and never dirties git while you browse. Optimised for Nicholas's own dev loop + a friend's — not the published-registry consumer story (publishing / former Item 27 stays deferred out of 1.0). Distribution remains the local-dep / clone path.

The target experience

# from anywhere inside your project
$ npx media-manager
› resolved root: ./src/assets/media_manager  (from media-manager.config.json)
› build/ missing — building once…  done (8.2s)
› ready on http://localhost:54731  (opening browser…)

Compare today: you must npm run build first (hard error otherwise), pass an explicit root every time, accept a hardcoded :3000 that collides with your dev server, wait on a blind 2-second timer for auto-open, and watch git status go dirty just from opening a committed workspace.

Item 30 — Config discovery + CLI verbs

files: bin/media-manager.js · src/lib/storage/paths.ts

The config file

// <repo>/media-manager.config.json
{
  "root": "./src/assets/media_manager"
}
  • Single required field root. Validated on load — a missing/empty root, or a path that doesn't exist, produces a clear message (not a stack trace), routed through the same copy as doctor.
  • root is resolved relative to the config file's directory, so running from a subdir still finds the right data. Room to add port / open defaults later; out of scope now.

Resolution precedence

1. explicit positional arg      media-manager ./some/root
2. MEDIA_MANAGER_ROOT env       (test:serve, muscle memory)
3. media-manager.config.json    walk up from cwd; root relative to the file
4. → friendly doctor-style error

Arg and env keep winning so the existing node bin/media-manager.js <root> and npm run test:serve flows are untouched.

Verbs

serve default

Current behaviour + the new discovery/build/port logic. Bare media-manager ⇒ serve.

init

Scaffold an empty workspace by reusing the app's first-launch healing (don't hand-roll the seed). Writes a starter media-manager.config.json if absent.

doctor first-class

Diagnoses the launch without starting the server — what an unfamiliar user (your friend) hits first.

$ media-manager doctor
✔ config        ./media-manager.config.json
✔ root          ./src/assets/media_manager  (exists, writable)
✔ workspace     media/ hub + 2 classes, 3 record types
! build         build/ missing — will build on first serve (or run --rebuild)
✔ port          ephemeral (OS-assigned)

Failure mode mock — the friendly error path (precedence step 4) shares this renderer:

$ media-manager
✘ Could not find a workspace.
  Tried: no path arg · MEDIA_MANAGER_ROOT unset · no media-manager.config.json found
  (searched . up to /home/you)

  Fix one of:
    • run media-manager ./path/to/data
    • drop a media-manager.config.json with { "root": "…" }
    • run media-manager init to scaffold a fresh workspace

Item 31 — Ephemeral port + accurate auto-open

files: bin/media-manager.js

Spike result (adapter-node 5.5.2). PORT=0 is accepted — it's passed straight to server.listen({ port }), so the OS assigns a free port. But the adapter's "Listening on" log echoes the requested value (prints :0), and our bin runs the server as a child process (spawn(node, [build])) — so the real bound port is reachable neither from stdout nor from the JS exports. The brief's assumed "parse stdout" fallback doesn't work as-is.

Chosen approach — parent-side probe

  1. In the bin, open a throwaway listener on :0, read server.address().port, close it.
  2. Spawn node build with PORT=<probed> (still a child, minimal change).
  3. Now the child logs the real port — so readiness = watch child stdout for the "Listening on" line, replacing today's blind setTimeout(…, 2000). Open http://localhost:<probed> the instant it appears.
  4. Keep --port N to pin; --no-open still suppresses the browser.

Caveat: a tiny TOCTOU window between closing the probe and the child binding — negligible for a local single-user tool, standard practice. Race-free alternative (deferred): stop spawning, run the server in-process via import('./build/handler.js') → http.createServer(handler).listen(0) and read address().port directly. Bigger lifecycle refactor; revisit only if the race ever bites.

Item 32 — Quiet heal (browse without dirtying git)

files: src/lib/storage/manifest.ts · src/lib/storage/classRepo.ts (+ manifest.test.ts)

Spike result — smaller than the brief assumed. The no-op write guards already exist: reconcile (manifest.ts:307), applyMembershipIndex (manifest.ts:188) and setEntryDimensions (manifest.ts:213) each write only inside an if (changed). So a fully-consistent committed workspace very likely already browses git-clean. Item 32 is targeted fixes, not a rewrite.
Build-time finding (2026-06-28) — the content-hash trigger is reclassified as deferred perf, not a 1.0 fix. Key fact: git tracks content, not mtime. So the mtime-resync re-firing after a git checkout writes nothing (the applyMembershipIndex guard holds) and, even if it rewrote identical bytes, git would still show clean. ⇒ The git-clean goal is already met by the existing guards; the content-hash trigger would only save some class-file re-parsing (CPU), at the cost of fragile signature-sync across every membership write path. Decision: ship Item 32 as the test guarantee (added — manifest.test.ts "quiet heal" block: no-change ⇒ no-write, real-change ⇒ write + report, stays-missing ⇒ no further write) plus the empirical git status-clean gate; defer the content-hash trigger to a small post-1.0 perf item.

Remaining churn vectors & the fixes

VectorWhat happensFix
mtime trigger (load-bearing) reconcileAndResync (classRepo.ts:738) marks the index "stale" when any class-file mtime > manifest mtime. git checkout doesn't preserve mtimes — it stamps all files at checkout — so a class file often reads as "newer," firing the resync on every browse. It won't write (the guard holds) so it's CPU churn, not git churn — but it means we can't rely on mtime once writes are fully conditional. Replace the mtime gate with a content-hash trigger: store a hash of the class-membership inputs in the manifest; resync only when the hash differs. Stable across checkout.
Dims backfill backfillDimensions (classRepo.ts:782) measures & persists width/height on the first browse of any workspace whose committed manifest lacks them — real new data, but it dirties git on first open. Decision (default in this plan): accept it — it's genuine healing and quiesces after one browse. If we want zero first-open churn, gate persistence behind a future read-only flag. (Flagging for your call.)
Equivalence (OQ3) Guards compare structured fields, while writes serialise the Zod-normalised object — fine today, but a future guard that compared serialised bytes could rewrite on pure formatting drift. Keep structural equivalence; add one belt-and-suspenders check — compare the next manifest against the on-disk parsed manifest before writing, so normalize-on-read can never trigger a write.
Still owed: the empirical gate. "Browse a git-committed workspace ⇒ git status clean" needs build + test:serve against a committed copy of test-fixtures/ — the headline acceptance for Item 32. Not run yet (kept clear of the tree while another agent was editing). It may already pass for a consistent fixture; if it does, Item 32 collapses to the content-hash trigger + the structural belt.

On-disk / surface changes (summary)

  • New file consumers read: media-manager.config.json at/above cwd (optional; never written by serve, written by init).
  • Manifest schema: add a small membership-hash field (Item 32 content-hash trigger). Backward-compatible default; absent ⇒ treated as stale once, then written.
  • No HTTP API changes. Purely launcher + storage-internal.
  • Docs: CLAUDE.md "Commands"/launch section (verbs + discovery + ephemeral port; PORT no longer the contract); FEATURES.md note on quiet-heal; fixtures only if the manifest gains the hash field (regenerate seed).

Cut line (explicitly out of 1.0)

  • Publishing (npm bin/files/engines, prepublish build, scoped name, native-dep cross-platform audit — former Item 27). Local-dep / clone only for now.
    Forward note — consumers won't build. When this is published, a prepublishOnly build runs once at publish time and build/ ships in the tarball (via the files whitelist), so npx media-manager runs the pre-built server with no build on the user's machine. The build-on-demand we add now is the bridge for the un-published (git-clone / local-dep) era — once build/ ships in the package it's simply present and the on-demand path goes dormant. The real "works for someone else" risk is install-time, not build: exiftool-vendored / heic-convert resolve platform binaries on npm install — hence the deferred cross-platform native-dep audit.
  • Reader / Mode B (Item 33) — separate brief, lands after this trio.
  • Config schema beyond root (port/open defaults) — easy to add later.
  • In-process server (the race-free port alternative) — only if the probe's TOCTOU ever bites.
  • Read-only mode for suppressing dims-backfill persistence — only if we reject the "accept first-open dims write" default above.
