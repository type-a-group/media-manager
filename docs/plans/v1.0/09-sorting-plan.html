Item 9 — Sorting & Ordering

media-manager 1.0 · Phase 2 design plan · ready · size M

Backlog: Item 9 · Brief: docs/plans/v1.0/09-sorting.md · This plan supersedes the brief where they differ (agile rule).

One-line goal. Add a server-side sort field + direction control to every list view (Files hub · single-class catalog · Records), composing with the existing search + filter + group-by, with the choice persisted durably per entity.

1 · Decisions resolved (Phase 1 interview)

DecisionResolution
PersistencePer-entity, durable — each class / record-type / the Files hub remembers its own sort, like group-by and title-by do today.
Control locationContent toolbar, next to group-by and grid-size (where view controls already live). Not in the rail.
Default sortMost recent first — last_modified desc where a record exists; on the Files hub (no record context) this resolves to created_at desc.
Empty valuesAlways last — empties sink to the bottom in both directions (never crowd the top), tie-broken by id.
Direction controlSeparate toggle button (↑/↓ icon) beside the field Select — direction stays visible and one-tap.
Folded-in cleanup (this item). While we add sort-by-recency, surface the last-modified timestamp on every file/record so the ordering is legible — see §7. Bundling it here keeps the recency UX honest: no point sorting by a date the user can't see.

2 · Where each sort lives (persistence)

Three surfaces, three existing per-entity stores. No new storage file; each gains two optional keys: sortField + sortDir.

SurfacePersisted inWrite path
Records (/media/[typeId]) MediaTypeSettingsFile → type settings.json POST /api/media-types/[typeId]/settings
Single-class catalog (Files, one class) ClassConfig → media/classes/<id>.json PATCH /api/classes/[id]
All Files hub (no / multi class) MediaSettings → media/settings.json POST /api/settings

Mirrors how displayField/subtitleField/gridGroupByField persist per-entity and how classOrder lives in media settings. Globals is a singleton with no list → no sort control.

3 · Sortable fields per surface

Built-in keys are recognized by the server; schema fields are offered from schemaUserFields(). A surface only offers fields it actually has.

Records

Name / titleLast modified …each user schema field

Sorts title_value for Name (so a chosen title-by wins, consistent with the grid label).

Single-class catalog

Name (filename)Last modified Date addedSize …each class schema field

last_modified comes from the class record; created_at/size from the manifest entry.

All Files hub

Name (filename)Date addedSize

No single class → no schema fields and no per-record last_modified. The "most recent" default resolves to created_at desc here.

Built-in key vocabulary (server contract). name last_modified created_at size — plus a raw <schemaFieldKey> for any user field. Unknown/empty sort ⇒ the surface default.

4 · Sort semantics (one comparator, shared)

  • Strings (name, string/url/list fields): case-insensitive localeCompare. URL → display_name; list → joined text; boolean → false < true.
  • Numbers (size, number fields): numeric.
  • Dates (last_modified, created_at — ISO strings): lexicographic compare (correct & stable for ISO-8601).
  • Empties last, always. Missing / null / "" sort to the end regardless of direction — incomplete rows never crowd the top.
  • Stable tie-break by id so equal keys keep a deterministic order across reloads.

Lives as one helper (e.g. core/sort.ts client+server safe) reused by all three repos so behaviour is identical and unit-testable in isolation.

5 · API changes

Add ?sort=<key>&dir=<asc|desc> to three list endpoints; apply in the repo, replacing the hardcoded file_name sort.

EndpointRepo function
GET /api/media-types/[typeId]/records (list)jsonRepo.listRecords
GET /api/filesclassRepo.listAllFiles
GET /api/classes/[id]/membersclassRepo.listClassMembers

Both classRepo functions currently end with files.sort((a,b) => a.file_name.localeCompare(b.file_name)) — that line becomes sortItems(files, sortField, sortDir). listRecords currently returns disk order; it gets the same call before projection.

Settings endpoints extended: PatchSchema in /api/settings gains sortField?/sortDir?; MediaTypeSettingsFile and ClassConfig Zod schemas + their PATCH/POST handlers gain the two keys (preserved on write, like the other per-entity prefs).

6 · UI — the toolbar control

A shadcn Select (field) + an icon Button (direction toggle), placed beside the existing Group-by / Size controls in each page's content toolbar.

Files hub — content toolbar ┌────────────────────────────────────────────────────────────────────────────┐ │ Group by [ Classes ▾ ] Sort [ Date added ▾ ] [ ↓ ] Size [ ▭ ▭▭ ▭▭▭ ] │ └────────────────────────────────────────────────────────────────────────────┘ ▲ ▲ field Select direction toggle (↓ desc / ↑ asc) Single-class catalog (class "images" selected) ┌────────────────────────────────────────────────────────────────────────────┐ │ Group by [ None ▾ ] Sort [ Last modified ▾ ] [ ↓ ] Size [ ▭ ▭▭ ▭▭▭ ] │ └────────────────────────────────────────────────────────────────────────────┘ Sort field options: Name · Last modified · Date added · Size · {schema fields} Records (/media/notes) ┌────────────────────────────────────────────────────────────────────────────┐ │ Group by [ None ▾ ] Sort [ Name ▾ ] [ ↑ ] + New record Settings │ └────────────────────────────────────────────────────────────────────────────┘ Sort field options: Name · Last modified · {schema fields}
  • Direction toggle = an icon button cycling asc⇄desc (ArrowDown/ArrowUp from lucide), with an aria-label announcing the current direction.
  • Changing either control re-runs the list query with the new ?sort/?dir and persists via the surface's write path — same pattern as changeCatalogGroupBy already uses for group-by.
  • Empty / loading states: control is enabled whenever the grid is; switching sort shows the grid's existing loading affordance. No special empty-state — sort of zero rows is a no-op.
  • Mobile: same controls; they wrap under the toolbar like the existing ones (no new responsive work beyond matching group-by).

7 · Surface the last-modified timestamp (folded-in cleanup)

Today the timestamp is stored but never shown. Make it visible in the side panel only, so sort-by-recency is legible without cluttering the grid. One shared formatter; no new endpoint.

What date each editor shows

EditorField shownLabel
RecordDetailPane (records)record last_modified"Modified …"
FileEditorPanel (blob)blob created_at"Added …"
FileEditorPanel per-class sectionthat class record's last_modified"Modified …"
GlobalsEditorPanerecord last_modified"Modified …"

No new data wiring: records already carry last_modified; the blob's created_at is on FileItem; and FileEditorPanel already loads each class section's record (which carries last_modified). Missing timestamps render nothing.

Where it renders

  • Side panel only — a muted "Modified …" / "Added …" line in the shared EditorPanelShell header area, so it shows for FileEditorPanel, RecordDetailPane, and GlobalsEditorPane consistently (the CLAUDE.md parity rule). Not on the grid tiles — the tile stays clean.
  • Formatter — one formatTimestamp(iso) helper in core/ client+server safe: absolute local date+time (e.g. Jun 26, 2026, 3:40 PM) with the full ISO string as a title tooltip. No relative-time dependency for 1.0.

9 · Verification gates (Phase 4)

  • npm run check + npm run lint clean on touched files.
  • npm run test — comparator: each built-in sort, asc/desc, empties-last, stable id tie-break, every field type (string/number/boolean/list/url/file). Plus a per-repo test asserting the list applies it and composes with search/filter.
  • npm run test:serve — sort by name / recency / a schema field in Files, a single-class catalog, and Records; confirm it stacks with an active search + filter + group-by.
  • Persistence across reload — set a non-default sort per surface, reload, confirm it sticks (and is independent per entity).
  • UI capture (test-ui-feature) of the control + before/after ordering for at least Files and Records, and the side-panel timestamp line.
  • FEATURES.md updated (new query params); Item 9 moved to Shipped & folded; triage HTML count synced.

10 · Explicit cut-line (out of scope for 1.0)

  • Manual drag-reorder / arbitrary saved orderings — sorting is field-driven only.
  • Multi-key sort (primary + secondary chosen by the user) — single key + id tie-break only.
  • Per-group sort inside group-by buckets beyond the single global comparator.
  • Sort on Globals — singleton, no list.
