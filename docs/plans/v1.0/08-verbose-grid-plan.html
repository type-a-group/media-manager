Item 8 — Configurable & Verbose Grid Display

1.0 feature plan · Phase 2 — for approval · usefulness 5/5 · backlog Item 8 · brief 08-verbose-grid.md

Today a tile shows a thumbnail + one title line. This adds an opt-in verbose mode: a Lightroom-style card that shows a chosen set of metadata fields under the thumbnail, so a catalog can be scanned by its data, not just its pictures. All three Phase-1 open questions are resolved (below); this plan is the design to approve before any code.

1 · Decisions (Phase-1 interview — resolved)

QuestionDecisionConsequence
Layout Masonry tiles, fields stacked below Stays in the existing balanced Masonry grid. Verbose cards are taller → variable height, which masonry already packs. No new layout mode.
Persistence Durable per-type default Verbose on/off + chosen fields saved per entity (class config / record-type settings), exactly like group-by & title-by. Remembered across reloads.
Field cap Hard cap 6 Picker disables further selection at 6. Keeps cards readable + list payloads lean. No virtualization in 1.0.
Scope Both Files + Records Built once in the shared DataGrid via the side-agnostic GridItem contract; both hubs get it.

Where verbose applies

ViewFields offeredPersisted in
Records /media/[type]type schema user-fields (schemaUserFields)settings.json (type)
Files — single-class catalog ?class=that class's schema user-fieldsclass config
Files — All Files / multi-classintrinsic only: size · dimensions · type · createdmedia/settings.json

Rationale: schema-field verbose only makes sense where there is one schema. All Files holds heterogeneous blobs, so it offers the small intrinsic set (already on FileItem — no extra fetch). This keeps the whole "Files hub" covered without inventing cross-class fields.

3 · Data / API / on-disk changes

The load-bearing change: fields come back inline on each list row

No per-tile fetch. The list endpoints accept a requested field set and return stringified values per row, the same way title_value already works.

LayerChange
GridItem (data-grid/types.ts) Add fields?: { label: string; value: string }[] — already label-resolved & stringified by the host adapter. The grid renders these rows under the label in both variants; absent/empty ⇒ unchanged compact tile.
JsonListItem & FileItem (core/types.ts) Add field_values?: Record<string, string> — the requested keys → stringified value ('' when empty, so rows stay aligned tile-to-tile). Derived, not persisted; absent unless requested.
Records list (jsonRepo.listRecords) Accept fields: string[]; for each row, stringifyFieldValue each requested key into field_values. Capped at 6 server-side (defensive). Endpoint param ?fields=camera,iso,lens.
Class members (classRepo.listClassMembers) Same: accept requested fields, stringify from the member record into field_values. Single-class catalog only (mirrors how title_value is already scoped there).
Per-type / per-class config Add verbose?: boolean + verboseFields?: string[] to ClassConfigSchema, MediaTypeSettingsFile, and the All-Files media/settings.json. Saved via the existing config-patch / settings-POST paths (no new endpoints).

No migration

All new fields are optional and absent ⇒ today's behaviour (compact). Existing data roots and the test-fixtures/ seed read identically; the fixture only needs a class flipped to verbose to exercise the feature in test:serve.

4 · Build slices (Phase 3)

Smallest end-to-end paths first; each ships behind its own verification gate.

  1. Slice 1 — data spine. field_values on the two list endpoints + a ?fields= param; unit-tested. No UI yet. Proves the inline-values shape on both sides before any rendering.
  2. Slice 2 — render. GridItem.fields + key/value rows in DataGrid; host adapters map field_values → fields. Verbose forced on via a temporary constant to eyeball it.
  3. Slice 3 — Records control + persistence. Toolbar Fields popover (switch + capped picker) on /media/[type], persisted to type settings; reload-survives.
  4. Slice 4 — Files control + persistence. Same popover on the single-class catalog (class config) and All-Files intrinsic set (media/settings.json).

5 · Verification gates (Phase 4)

  • npm run check + npm run lint clean for touched files.
  • npm run test — new vitest: list endpoints return correct field_values for requested keys, empty ⇒ '', the 6-field cap is enforced, and no per-record fan-out (one list call). Add to classRepo.test.ts + the records repo test.
  • npm run test:serve — pick a field subset + toggle verbose on Files (catalog) and Records; confirm it composes with sort / filter / group-by / size and persists across reload. Test at the 6-field ceiling.
  • UI capture — compact vs verbose at two grid sizes, image tiles + text tiles (via scripts/ui-capture.mjs).
  • Docs same change — FEATURES.md (new list params + grid mode), test-fixtures/ seed if a class default flips, Item 8 → Shipped & folded, triage HTML synced.

6 · Cut line (explicitly out of 1.0)

  • Virtualization / windowing — large-catalog scroll perf is its own concern (tile count, not fields). Carved out as Item 40 in the backlog; pull in only if real catalogs feel slow.
  • List/row view (fields beside a thumbnail, table-style) — a different layout mode; not built. Masonry-with-fields-below covers the scanning use case for 1.0.
  • Per-session override on top of the durable default — durable per-type only; no transient layer.
  • Cross-class fields in All Files / multi-class — intrinsic set only there; no schema-field union.

7 · Hand-off smoke checklist (after build, against npm run test:serve)

  1. Open a class catalog (e.g. ?class=images), open Fields ▾, toggle Verbose, check 3 fields → tiles grow key/value rows.
  2. Reload → verbose + the same 3 fields are still on (durable per-class).
  3. Open a different class → it's back to its own setting (independent per-entity).
  4. Try to check a 7th field → it's disabled at 6.
  5. Repeat on a Records type; confirm group-by + sort + size still work alongside verbose.
  6. All Files → Fields offers intrinsic only (size/dimensions/type/created), persists in its own scope.
