Item 13 — Dimension Consistency Check & Fix

1.0 feature plan · file-first · size S no new dependency phase 2 — needs approval

1 · What we're building (and what changed from the brief)

The original brief framed Item 13 as a free-floating "swap width/height" button. Per the interview, that's out. Swapping should only surface when the stored dimensions actually disagree with the real image — it's a metadata-vs-photo inconsistency tool, not a thing you reach for casually.

So the feature is now:

  • Detect when a blob's stored manifest dimensions don't match the real image (the orientation-corrected dimensions read from the file).
  • Surface it in the Metadata popup, with an external warning badge on the metadata button so you notice without opening it.
  • Offer two one-click fixes: a smart "Correct dimensions" (writes the orientation-aware real dims) and a manual "Swap W↔H" escape hatch.

Resolved open questions

Question Decision
Manual swap only, or smart + manual? Smart "Correct dimensions" + manual "Swap W↔H" escape hatch. Both dependency-free.
Where does it live? Metadata popup, with an external warning badge on the metadata button (only when a mismatch is detected).
Also bake/normalize EXIF orientation? No — dims only for 1.0. Physically rotating pixels needs sharp (Item 35) and is deferred.
Which width/height is corrected? The intrinsic dims in media/manifest.json (the single source of truth; class catalog rows just project it).

2 · Why dims get transposed (the mechanism)

Almost always EXIF Orientation. A phone stores the pixels in one orientation and writes a rotation flag. The browser respects the flag and displays the photo rotated, but our manifest stored the pre-rotation numbers. So the manifest says 4000×3000 while the photo visibly reads 3000×4000 — which would also make a masonry tile (Item 34) the wrong shape.

ExifTool gives us ImageWidth, ImageHeight and Orientation with zero new dependencies. The display dims are the file's raw dims, swapped when orientation is a quarter-turn: 

Orientation 1  normal              → keep  (w, h)
Orientation 2  flip-horizontal     → keep  (w, h)
Orientation 3  180°                → keep  (w, h)
Orientation 4  flip-vertical       → keep  (w, h)
Orientation 5  transpose           → SWAP  (h, w)
Orientation 6  rotate 90° CW       → SWAP  (h, w)
Orientation 7  transverse          → SWAP  (h, w)
Orientation 8  rotate 90° CCW      → SWAP  (h, w)

So corrected = orientation ∈ {5,6,7,8} ? (rawH, rawW) : (rawW, rawH). A mismatch = the manifest's stored dims differ from corrected. This also catches plain stale dims (file edited externally), because corrected is re-read live from the file.

3 · UI

3a · Metadata button — warning badge (no popup open)

File editor panel actions: ⓘ! ← amber dot when a dimension mismatch is detected

Tooltip changes from "View file metadata" to "View file metadata — stored dimensions look wrong" when flagged. The badge is absent for files with no mismatch (the common case) and for non-images.

3b · Inside the popup — warning banner + fixes

⚠ Stored dimensions don't match the image
Stored: 4000 × 3000  ·  Image (orientation-corrected): 3000 × 4000  ·  EXIF Orientation: 6
Correct dimensions → 3000 × 4000 Swap W↔H

Mirrors the existing "Fix extension" banner exactly (same placement, same button idiom). Disappears after a successful fix; the "Image Properties" width/height rows below it update in place.

3c · No mismatch (the common case)

No badge on the button, no banner in the popup — the dialog looks exactly as it does today. The feature is invisible until there's something to fix.

4 · Data flow

panel mounts a file→ GET /api/files/[id]/dimension-check → badge if mismatch
open popup→ GET /api/files/[id]/metadata (already carries the check) → banner + fix buttons
click a fix→ POST /api/files/[id]/dimensions { width, height } → manifest updated→ refetch + onchanged()

5 · Implementation — files & changes

Layer File Change
core (client+server) src/lib/core/images.ts Pure helpers orientationSwapsDimensions(o) + orientationCorrectedDimensions(w, h, o). Single source for the rule. New unit tests in images.test.ts (or a small new test file).
server src/lib/server/fileMetadata.ts dimensionConsistency(filesDir, filename, stored): read exiftool w/h/orientation, compute corrected, return { stored, fileRaw, orientation, corrected, mismatch }. Reused by both endpoints. PDFs / unreadable dims ⇒ mismatch:false.
storage src/lib/storage/manifest.ts getEntryDimensions(id) (read stored w/h) + setBlobDimensions(id, {width,height}) (locked write of explicit dims). Lazy backfill already skips entries that have dims, so corrected values persist.
storage src/lib/storage/classRepo.ts Thin re-exports / wiring if needed (getFilenameForId already exists). No change to listAllFiles shape — FileItem.width/height already present.
API src/routes/api/files/[id]/dimension-check/+server.ts (new) GET → dimensionConsistency for the badge (one lightweight read). 400 on bad id, 404 if no blob.
API src/routes/api/files/[id]/dimensions/+server.ts (new) POST { width:number, height:number } → setBlobDimensions. Validates positive integers. Returns { success, width, height }.
API src/routes/api/files/[id]/metadata/+server.ts Attach dimensionCheck to the response (computed from the manifest stored dims + the metadata it already read — no second exiftool read).
client src/lib/api/files.ts apiCheckFileDimensions(id) + apiSetFileDimensions(id, w, h).
component src/lib/components/MetadataButton.svelte On mount (image only): fetch the check → badge + tooltip. In dialog: warning banner + "Correct dimensions" / "Swap W↔H" handlers (model on the existing handleFixExtension). After a fix: refetch + onchanged() so the grid tile / panel reflect corrected dims.

On-disk / schema impact

  • No schema change. manifest.json entries already have optional width/height; we just write corrected values.
  • No migration. Existing data is untouched until a user clicks a fix on a specific file.
  • Fixture: add one deliberately-transposed image (or set wrong stored dims on an existing fixture blob) to test-fixtures/media/ so the warning path is exercisable in test:serve.

6 · Edge cases & decisions

Case Behavior
Non-image (PDF, txt) No check, no badge. dimensionConsistency returns mismatch=false.
Stored dims absent (never backfilled) Not a "mismatch" to fix — the normal lazy backfill already writes correct-on-first-read dims. We only flag when stored dims exist and differ from corrected.
File unreadable by exiftool mismatch=false (fail safe — never show a bogus warning).
"Swap" when there is no orientation reason Still allowed (manual escape hatch) — it just transposes whatever is stored. That's the point of the manual option.
Fix is reversible No confirm dialog — swapping again / re-correcting undoes it. Keep it one click.
