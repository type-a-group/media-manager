Item 18 — Records Storage Reorganization

1.0 storage plan · file-first size M no new dependency migration (explicit, no dual-read) phase 2 — needs approval

1 · What we're building (and what changed from the brief)

The json-kind record side still lives in scattered top-level per-type folders (<root>/notes/, …) — a leftover from before the file-first media/ reorg settled. This item gathers them under a single top-level records/ folder, and folds in two adjacent storage warts found in the interview sweep. It is a pure refactor / migration — nothing user-facing changes. All risk is in migration correctness.

The brief asked for "records reorg + any other storage organization." The interview resolved its three open questions and swept for adjacent warts; the actual reorder UX that was briefly considered (drag/move classes & types) was split out as a feature — see the cut line.

Resolved decisions

Question Decision
Where does globals fit? Stays top-level (<root>/globals/). It's the reserved json singleton, already resolved via a hardcoded path.join(rootDir, 'globals') in mediaTypes.ts + RESERVED_TYPE_IDS. Keeping it top-level matches that special-casing.
Per-type subfolders & settings split? Each type is its own subfolder records/<typeId>/{settings.json, data.json}. A new records/settings.json is created now for cross-type prefs, carrying a dormant typeOrder (latent parity with media's dormant classOrder). Per-type settings.json keeps its own keys.
Migration & backward read? Explicit upgrade-data only (dry-run default, --apply), like the file-first migration. No dual-read — the app reads only the new layout and fails loudly ("run upgrade-data") on the old one. One layout to reason about at runtime.
Sweep: media/settings.json is mislabeled It's actually app-wide settings (gridSize, autosave, rail, sort, verbose) plus the one media key classOrder. Hoist app-wide keys to a new <root>/settings.json; media/settings.json keeps only classOrder.
Sweep: dead dataFileName field Every type's settings.json carries dataFileName, always 'data.json', never set otherwise. Drop it; hardcode data.json in paths.ts.
Cut line (explicitly NOT in this item)
  • File-first media/ internals — untouched.
  • No reorder feature — only the dormant typeOrder field. Activating classOrder/typeOrder (reorder UI + sort-on-read) is Item 41, not 1.0.
  • No dual-read / backward-compat reading of the old layout.

The single biggest behavioural detail: settings split across three files by scope, each with one owner. /api/settings keeps the same request/response shape — it just backs a different file on disk.

4 · Discovery & resolution (paths.ts)

The two resolution helpers change; everything else flows through them.

  • listMediaTypeIds() — today scans the root and skips media/. After: scan records/ for type subfolders, then fold in the reserved top-level globals so it still appears as a type to the records APIs.
  • getMediaTypePaths(typeId) — resolve <root>/records/<typeId>, except typeId === 'globals' → <root>/globals (the one special-case). dataFileName indirection removed — dataPath is always <baseDir>/data.json.
  • New: getAppSettingsPath() → <root>/settings.json; getRecordsSettingsPath() → <root>/records/settings.json.
  • Reserved folder names: media, records, globals can't be user typeIds (creation guard) — prevents a type colliding with a structural folder.

5 · Migration (scripts/upgrade-data.mjs)

A new step, dry-run by default, --apply to write — mirroring the existing file-first migration discipline. Idempotent: running twice is a no-op.

1 · detect flat layout→ 2 · move type dirs → records/→ 3 · strip dataFileName→ 4 · split media/settings.json→ 5 · create records/settings.json
  1. Detect: any top-level dir other than media/, globals/, records/ that holds a json media-type settings.json = a type to move. (Also: media/settings.json that still has app-wide keys = needs the split.)
  2. Move: for each such type, move <root>/<typeId>/ → <root>/records/<typeId>/ (create records/ first).
  3. Strip: rewrite each moved type's settings.json without the dataFileName key (and rename a non-default data file to data.json if one ever existed).
  4. Split settings: read media/settings.json; write the app-wide keys to <root>/settings.json; rewrite media/settings.json with only classOrder (drop it entirely if absent).
  5. Create records/settings.json as {} if missing. globals/ is left exactly where it is.

All writes atomic (writeJsonFileAtomic); directory moves via fs.rename with a copy-fallback across devices. Dry-run prints the planned tree diff without touching disk.

Old-layout detection guard (no dual-read)

Because the app deliberately doesn't read the old layout, it must fail clearly rather than silently show an empty/half data root. At list time (the first listMediaTypeIds() / startup path), if a flat-layout signal is present — a top-level non-reserved json-type folder, or media/settings.json still carrying app-wide keys — throw: 

Old data layout detected at <root>.
Run:  npm run upgrade-data -- <root> --apply
(then restart). See docs/plans/v1.0/18-records-storage-reorg.md

The guard is read-only — it never mutates. A correctly-migrated root has no flat-layout signal, so the guard is invisible in normal operation.

On-disk / behaviour impact

  • Breaking on-disk change — but gated behind the explicit upgrade-data step; no auto-migration on first run (only a clear error).
  • No API shape change — endpoints and client wrappers are untouched; only the bytes-on-disk move.
  • Lock order & lazy-heal preserved — manifest-before-class unchanged; the records side gains no new locks.

7 · Edge cases & decisions

Case Behaviour
App run against an un-migrated (flat) root Guard throws the "run upgrade-data" error. No partial read, no mutation.
upgrade-data run twice Idempotent no-op — detection finds no flat-layout signal the second time.
A pre-existing type literally named records or media Migration aborts with a named conflict error (rename the type first). New creates are blocked by the reserved-name guard.
media/settings.json missing entirely App-wide settings fall back to defaults; <root>/settings.json created with defaults on first write. No classOrder ⇒ no media/settings.json needed.
globals/ present (the common case) Left in place; still resolved by the typeId === 'globals' special-case.
Empty data root (fresh install) No flat signal ⇒ guard passes; ensureGlobalsGroupExists + lazy creation produce the new layout directly.
