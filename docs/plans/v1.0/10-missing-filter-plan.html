Item 10 — Filter for Missing / Empty Fields

media-manager 1.0 · Phase 2 design plan · ready · size M

Backlog: Item 10 · Brief: docs/plans/v1.0/10-missing-filter.md · This plan supersedes the brief where they differ (agile rule).

One-line goal. Add an “Incomplete only” toggle plus a per-field “<field> is empty” quick-filter to the rail of every schema-bearing list view (Records type · single-class Files catalog), composing server-side with the existing search + filters + sort.

1 · Decisions resolved (Phase 1 interview)

DecisionResolution
Filter shapesBoth — an “Incomplete only” toggle (matches any record with ≥1 empty user field) and a per-field “<field> is empty” picker.
SurfacesRecords (per-type schema) + single-class Files catalog (class schema). The multi-class / All-Files hub gets nothing — no single schema to define “a field” against.
PlacementRail filter section — below the search box (Records), and below the class multi-select (Files catalog).
“Empty” definitionReuse the existing predicate verbatim: '' · null · undefined · empty list · url-object with blank url. (No new semantics — same as today’s is_empty clause.)
PersistenceTransient view state (like search & the class multi-select), not a durable per-entity default like sort. Resets on type/class switch.
Cut line (out of 1.0)No “is not empty” quick-filter, no per-field operator builder (general multi-clause UI stays retired), no All-Files-hub aggregate, no URL deep-link of the filter state.

2 · The stale-premise correction (why this is a new control, not a tweak)

The brief & CLAUDE.md assume a “rail filter UI” / RecordFilterPanel to hang this off. It no longer exists — the rich multi-clause filter builder was removed in the unify redesign; it survives only in docs/archive/. Today the rails expose search + search-field scope (both sides) and class multi-select + Unclassified (Files only). No page builds FilterClauses through any UI.

So Item 10 introduces a focused, first-class empty-filter control — it does not revive the general filter builder. The server plumbing it rides on, however, already exists and is fully tested (see §3).

3 · What already exists (reuse, don’t rebuild)

  • Both list endpoints accept ?filters=<json array of FilterClause> and apply them server-side: /api/media-types/[typeId]/records/list → jsonRepo.listRecords; /api/classes/[id]/members → classRepo.listClassMembers.
  • Both repos evaluate is_empty / is_not_empty via an identical isEmpty() predicate (jsonRepo.ts:79, classRepo.ts:571).
  • apiListRecordsForType already forwards a filters param; apiListClassMembers accepts filters too.
  • schemaUserFieldKeys() / schemaUserFields() (core/fieldKeys.ts) is the single “which fields does this entity expose” enumeration — drives the per-field picker.
The one gap. The per-field “<field> is empty” shape maps 1:1 onto a single existing {field, operator:'is_empty'} clause — pure reuse. But the clause array is AND-only, so it cannot express “any of these fields is empty” (an OR). The “Incomplete only” toggle therefore needs a small dedicated server flag — ?incomplete=1 — rather than being expanded into clauses on the client. This keeps all filtering in the one server location (per the brief’s build note).

7 · Composition & edge cases

  • All three (search · incomplete · per-field-empty) are ANDed — turning on more narrows results.
  • “Incomplete only” + “status is empty” together is allowed (redundant-ish but valid: status-empty ⟹ incomplete, so the per-field wins).
  • A schema with zero user fields → “Incomplete only” matches nothing useful and the per-field picker is empty; show the group but the Select reads “No fields”. (Rare; acceptable.)
  • Switching record type / soloed class resets both controls (transient).
  • Result-count feedback: the grid already reflects the filtered set; no separate counter added for 1.0.
  • Files: when the user un-solos (back to 0/≥2 classes) the control unmounts and its state resets so a stale incomplete can’t leak into the hub query.
