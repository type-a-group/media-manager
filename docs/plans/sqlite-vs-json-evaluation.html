Planning · Decision Brief

JSON-on-disk vs SQLite

An honest evaluation of whether media-manager should migrate its on-disk data format from JSON files to SQLite — and a recommended middle path.

Prepared 2026-06-28 · branch global-files · for discussion, no code changes proposed yet

TL;DR — the recommendation

Don't make SQLite the source of truth. At this app's realistic scale (hundreds to low-thousands of records per entity), plain JSON is the correct, simpler choice and SQLite would be premature — it forfeits the git-diffable / human-readable story the npx-in-host-repos vision prizes, imposes native-module distribution pain (worst exactly under npx), and trades away the flexible heal-on-read behavior.

If and when scale or search demand it, adopt the hybrid the codebase already models: JSON files stay canonical and git-tracked; add a gitignored, rebuildable SQLite + FTS5 index for the hot query paths, kept fresh by the same mtime-gated resync already used for manifest.classes[]. The app already thinks this way — the manifest is a derived cache over the class files.

01How we use JSON today

There is no database. Every entity is a pretty-printed JSON file under one data root. Three things matter for this decision: entity-sets are monolithic (one file holds all records of a class), nothing is cached (every read re-parses from disk), and every write rewrites the whole file.

The clever part the app already gets right

manifest.classes[] is a derived membership cache. The class files are the source of truth; the manifest index is rebuilt from them via an mtime-gated resync (if any class file is newer than the manifest, re-read and rebuild). This is exactly the "canonical files + derived index" pattern that the recommended hybrid generalizes — the architecture is already shaped for it.

04The scale breakpoint — your key question

You asked "where does JSON stop scaling well?" Here is the honest answer, anchored to JSON.parse / serialize behavior and this app's whole-file-rewrite model.

For a personal metadata tool over media files, realistic collections (a few thousand files, a handful of classes) sit comfortably in JSON's sweet spot. SQLite's performance case only becomes forcing if collections reach tens of thousands of records, or if rich, ranked full-text search becomes a headline feature.

05Three options on the table

A · JSON-primary (today)

Files are canonical. No DB. Keep refining within the current model.

B · SQLite-primary

A .db file becomes the source of truth. JSON files go away (except raw blobs).

C · Hybrid — JSON canonical + derived SQLite index (recommended)

JSON files stay the git-tracked source of truth. A gitignored, rebuildable SQLite + FTS5 database is a derived query index over them, kept fresh by the existing mtime-gated resync. Backup is still "copy the folder"; a corrupt index is fixed by rebuilding from JSON; no user-facing migrations.

SQLite's genuine wins

  • Partial updates — no whole-file rewrite
  • Indexed filters / sort / group-by
  • FTS5 full-text search with BM25 ranking
  • Membership becomes a clean join table — the derived classes[] cache + resync code can disappear
  • ACID transactions replace the fragile stale-steal lock

What SQLite-primary would cost

  • Git-diffability & hand-editing — gone
  • Heal-on-read / migrate-on-read flexibility — replaced by migration scripts
  • "Drop a file in, edit a class by hand" — gone
  • Native-module distribution pain under npx
  • Does not lift the single-node ceiling (see below)
Myth to retire: SQLite does not solve "single-node only." Its locking is local-filesystem only — it makes intra-host concurrency nicer and ACID-safe, but multi-machine sync is still a CRDT / sync-engine problem (libSQL replicas, etc.), orthogonal to this choice. Don't adopt SQLite expecting multi-device sync.

07The diffability tradeoff — the deciding factor for you

You rated git-diffability "between very important and somewhat." That weighting is exactly why SQLite-primary is the wrong end of the spectrum. SQLite is genuinely hostile to git:

  • Byte-identical content can produce different binaries (VACUUM, page layout) — so even no-op commits churn.
  • Committed .db files give noisy, unmergeable binary diffs and bloat history.
  • Merge conflicts are unrecoverable — you get a corrupt DB, not a conflict you can resolve. This is the direct enemy of the npx-in-a-git-repo vision.

Mitigations exist (git textconv to show sqlite3 .dump, sqldiff, committing a SQL dump instead of the binary), but they make diffs readable without making the committed artifact mergeable — and once you're committing a text dump, you've reinvented "text is canonical," at which point JSON is simpler and already in place.

The mainstream 2026 answer is not "git-diffable JSON or SQLite performance." It's canonical human-readable files + a gitignored, rebuildable SQLite index — which is option C.

08The hybrid — recommended shape

JSON stays canonical and git-tracked. SQLite is a disposable index you can delete and rebuild at any time. The sync mechanism is the one the app already has.

Driver choice for the hybrid: prefer node:sqlite (built into Node, RC as of v25.7) over better-sqlite3 to avoid native-module / prebuilt-binary pain — the single biggest risk for an npx-distributed tool — at the cost of requiring a recent Node. Code against the synchronous better-sqlite3-style API so the driver is swappable. Always set PRAGMA journal_mode=WAL and a sane busy_timeout.

What the hybrid keeps that SQLite-primary loses: every operational advantage stays in the JSON column — backup is still "copy the folder," index corruption is recoverable by rebuilding, and there are no user-facing migrations because the index regenerates. That asymmetry is the whole argument.

Prevailing wisdom for "data the user owns and may want in git": keep the canonical form as human-readable files and treat any database as a derived index/cache. Apps that go DB-primary either don't expect hand-editing (Actual) or invest heavily to restore the ownership story (Logseq).

10Migration surface (if you ever did go SQLite)

Good news: the storage layer is well-isolated. Routes never touch the filesystem directly (except raw blob I/O and EXIF); they call repo functions. A SQLite backend could hide behind the same function names — listAllFiles, listClassMembers, updateRecord, getMediaTypeRepo(...).listRecords wouldn't change signatures.

What gets rewritten: essentially all of src/lib/storage/ — json.ts + lock.ts (replaced by transactions), most of manifest.ts (the classes[] cache + resync fold into a join table and largely disappear), classRepo.ts and jsonRepo.ts (the bulk of the work; the two duplicated evaluateClause implementations become one SQL WHERE-builder), and the directory-scan discovery in paths.ts becomes table queries. Raw blobs in media/files/, the filename guards, and EXIF handling stay untouched.

The leaks to fix are functions that return disk-shaped artifacts (Manifest, ClassFile, JsonDataFile). For the hybrid, the surface is much smaller — you add an index/query module and a re-index hook, without rewriting the canonical writers.

11Decision criteria

Keep JSON-primary when…

  • Entities are realistically hundreds–low-thousands of records (the likely reality)
  • Git-diffability / hand-editability stay top-tier values
  • Heal-on-read flexibility is worth keeping
  • You want the simplest npx distribution (no native module, no Node floor)
  • Search needs stay modest (substring / exact-field)

Reach for SQLite / the hybrid when…

  • A single entity heads toward tens of thousands of records
  • Autosave full-file-rewrite latency / RAM becomes user-visible
  • You want ranked FTS5 search as a real feature
  • You need fast aggregates / field-values / group-by without full scans
  • Concurrent-read-while-writing matters more than the coarse lock provides

When those triggers fire, the answer is the hybrid (introduce a derived SQLite index), not SQLite-primary — you get the indexed/ranked queries while keeping every diffability and operational advantage.

12Open questions for you (before any build phase)

  1. Realistic ceiling. What's the largest collection you actually expect — a few thousand files, or could a class plausibly hit tens of thousands? This single number decides whether the hybrid is "someday" or "soon."
  2. Is search a headline feature? If ranked full-text search across fields is something you want to sell, that pulls the SQLite/FTS5 index forward independent of scale.
  3. Node version floor. Are you willing to require Node 22+/24+ for the npx tool? That's what lets you use node:sqlite and dodge the native-module distribution risk entirely.
  4. Instrument first? The cheapest next step is to add timing around record counts and autosave serialize time, so the "~10k / sub-50ms" trigger is data-driven rather than guessed. Want that as a small first task?
