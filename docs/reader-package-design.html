01The mental model

The reader is a "load once, read many" data layer. It is a pure function of already-parsed JSON — no fs, no server, no network. The host's bundler (Vite) does the file reading and asset hashing; the reader does the joining and hands back flat, iterable items.

Every public method returns either a flat item (MediaItem for a blob, MMRecord for a record) or a fluent Collection of them. There are only four "kinds" of thing in a workspace, and one method each:

Concept What it is Reader entry
Blob A file on disk (image, pdf, …) with a stable id. mm.media(), mm.file(id)
Class Opt-in metadata schema layered over a set of blobs. mm.media('photos'), mm.classes()
Record type Fileless json rows (projects, trips…). mm.records('trips'), mm.record(id), mm.types()
Globals The single reserved key/value singleton. mm.globals()

02What load() actually does

This is the one piece of glue you write, and the one people get wrong. Here it is, annotated: 

const mm = MediaManager.load({
  // 1️⃣ every JSON file, PARSED. import:'default' → the parsed object,
  //    eager:true → inline it now (not a lazy () => import()).
  data:  import.meta.glob('$assets/media_manager/**/*.json', { eager: true, import: 'default' }),

  // 2️⃣ every blob, as a URL. query:'?url' → Vite copies + content-hashes
  //    the asset and hands back its final served URL string.
  files: import.meta.glob('$assets/media_manager/media/files/*', { eager: true, query: '?url', import: 'default' }),
});

Why two globs, and why literals?

Vite can only bundle files whose glob pattern is a string literal it can see at build time. A library buried in node_modules can't glob your source tree, so the patterns must live in your code. The reader compensates by being path-driven: it inspects each entry's path to decide if it's the manifest, a class, a type, globals, or an asset — so the snippet is identical for every host; only the $assets/media_manager prefix changes.

The ?url asymmetry

The data glob wants the parsed JSON. The files glob wants a URL string, so it alone carries query: '?url'. This asymmetry is the #1 setup mistake — see the warning below.

Silent failure
If the files prefix is wrong (or you forget ?url), the JSON still loads, the manifest still parses — but every image comes back src: null, missing: true, with no error thrown. Blank gallery, no clue. This is exactly the "needs an AI to debug" class of problem, and the reason §05 proposes optional dev-mode diagnostics.

One-time host setup this depends on: a $assets alias in svelte.config.js pointing at ./src/assets, and the workspace copied (or symlinked) under it. nicb.at does exactly this.

03The manifest × asset join

A blob's src is not stored anywhere — it's computed by joining two maps the reader builds at load. This is worth seeing because it explains both the ext-case tolerance and the missing flag.

The same join is why references are cheap: a file-field stores only an id, and fileById(id) is memoized — so mm.file(id) and a resolved record.file('cover') are the same object in memory.

04What we're doing right

This is not a mess at the architectural level. Several choices match what the best "content as data" tools converged on independently:

✓ Pure, build-time data layer

No fs/env/network. Correct for a static host — same philosophy as Astro Content Collections & Velite.

✓ Facade + fluent Collection

.where().sortBy().map() — a recognised, Laravel-Collection-style ergonomic. Not a query DSL to learn.

✓ Layout-aware, path-driven

Hosts never hard-code media/classes/x.json; a format move doesn't silently break consumers.

✓ Shared identity + never-throw refs

Memoized resolution; a dangling id yields null, never a broken <img> or a crash.

✓ Loud version guard

WorkspaceFormatError on an absent/old manifest — actionable message, not a silent-empty gallery.

✓ Single source of truth

Value normalization & reserved-key handling mirror the editor's core/, so it can't drift.

05Where it bites

Grounded in the real nicb.at consumer, ordered by how much pain it causes:

Friction Symptom in real code Status
Stringly-typed access r.field('name') as string on every read; no autocomplete on field names; typos → silent undefined. proposed §06
Silent-empty failures Wrong files prefix → all missing. Typo'd class id → empty. Typo'd key → undefined. No warnings. proposed §05
No cross-record refs record-type fields handed back a raw id; no resolver, unlike file/files. fixed §07
Record name collision Shadowed TS's built-in Record<K,V>; forced globalThis.Record internally, import { Record as MMRecord } in hosts. fixed §08
Two identical import paths media-manager/reader vs /reader/vite are byte-identical; "which do I use?" keep, document

Proposed: opt-in dev diagnostics

The silent-empty class of bug is the biggest "needs AI" driver. A cheap, non-breaking fix: a { dev: true } (or import.meta.env.DEV) mode that console.warns once for the tell-tale signs — an unknown class/type id passed to media()/records(), a blob whose asset didn't resolve, a load where all assets are missing (near-certain glob-prefix mistake). Zero cost in prod, turns "blank page" into a one-line message.

06Fixing "stringly-typed"

Today field() returns unknown, so the host casts everywhere. This is the single biggest ergonomic complaint, and it's exactly the problem Astro / Velite / Contentlayer exist to solve: schema → generated TypeScript types. Three tiers, increasing power and machinery:

Tier 1 — generic accessor no build step

Smallest change. Let the caller assert the return type at the call site instead of after it: 

// today
const name = r.field('name') as string;
// tier 1 — field<T>(key): T
const name = r.field<string>('name');

Cheap, honest (it's still an assertion), removes visual noise. Doesn't validate the key or the shape — a typo is still a runtime undefined.

Tier 2 — typed item shape no build step

Parameterize the query methods on a host-written interface, so the whole projection is typed and keys autocomplete: 

interface Project { name: string; date: string; tags: string[]; }

const projects = mm.records<Project>('projects')
  .sortBy('date', 'desc')          // 'date' checked against Project
  .map(r => r.field('name'));      // → string, no cast

You maintain the interface by hand (it can drift from the real schema), but there's still no build step and the ergonomics jump dramatically. A good default if we want to stay zero-build.

Tier 3 — schema typegen the Astro / Velite answer

The workspace already stores the schema in each class/type file. A media-manager typegen <root> command reads those schemas and emits a .d.ts that maps every class/type id to a fully-typed field shape — the same trick Astro's .astro/types.d.ts and Velite's generated .d.ts use. Then mm.records('projects') is typed with no hand-written interface and no casts, and 'projects' / 'name' both autocomplete.

Recommendation
Ship Tier 2 now (a small, backwards-compatible generic on records/media) — it removes nearly all the as noise with zero machinery — and treat Tier 3 typegen as the next backlog item once we want the full autocomplete-from-schema experience. Both are additive; neither breaks the current string-key API.

07Reference resolution shipped this pass

A workspace has two kinds of reference field: file (→ a blob) and record (→ another record, e.g. a project's lead → a people row). Only file was resolvable. Now they're symmetric:

File fields Record fields new
single item.file(key) → MediaItem item.record(key) → MMRecord
list item.files(key) → Collection item.records(key) → Collection
by id mm.file(id) mm.record(id)

Resolution is by id alone — record ids are globally unique across types, so the reader builds one id→record index at load (mirroring the manifest's id→blob index) and you never name the target type. Identity is shared and dangling ids drop, exactly like files. That means you can chain hops across the graph: 

const p = mm.records('projects').first();
p.record('lead')?.field('name');        // → "Ada"
p.records('contributors');                // → Collection<MMRecord> (dangling dropped)
p.record('lead')?.file('avatar')?.src;  // hop record → blob → url

08Naming: why MMRecord, not a blanket prefix shipped

The problem was one name

Record shadows TypeScript's built-in Record<K, V>. That forced globalThis.Record all over the reader internals and import { Record as MMRecord } in nicb.at. MediaItem and Collection collide with nothing.

The fix: rename the collider only

Record → MMRecord. This also let us delete every globalThis.Record (the built-in is unshadowed again), a net simplification. Blanket MM-prefixing (MMCollection, MMMediaItem) would only add noise for no collision — so we didn't.

Consumer migration
nicb.at currently does import { Record as MMRecord }. When it bumps to a build that includes this rename, that becomes a plain import { MMRecord } from 'media-manager/reader/vite'. (It pins a published tag, so nothing breaks until the bump.)

10Prior art

The patterns above aren't invented here — they're the settled answers from the "content as data" ecosystem:

  • Astro Content Collections — defineCollection + Zod schema → auto-generated TS types on getCollection(); the canonical schema-typed-query pattern, and the model for Tier-3 typegen. It also resolves cross-collection reference()s (our record/records) and typed asset helpers (our file()/src).
  • Velite — "type-safe data layer": Markdown/JSON → Zod-validated, emits JSON + a .d.ts, framework-agnostic. Closest living analogue to what the reader is; confirms the pure-data + generated-types direction.
  • Contentlayer (now unmaintained) — the originator of "content → typed data" for the JS ecosystem; Velite is its spiritual successor. A reminder to keep the reader small and dependency-light.
