Records Explorer

A dedicated three-pane workspace for /media records — a separate-but-related sub-app inside Media Manager. It deliberately does not mirror the Files hub: records aren't blobs, so they get a layout built for browsing structured rows, not a wall of thumbnails.

Mental model

Files and Records are two sibling apps under one roof. Files = blobs + opt-in classes, shown as a thumbnail grid. Records = standalone schema-backed types, shown as an explorer: type rail → record list → inline detail. The rail is the cross-cutting navigator (the thing you liked from /files), but it lives only here and switches the active record type.

What it reuses vs. builds

FieldInput Reused as-is — renders every field type in the detail pane (the shared schema-driven input).
RecordFilterPanel Reused, relocated into the list-column "Filter ▾" popover instead of a full sidebar.
SchemaEditorButton · RecordBulkActions Reused — schema editor (🗄) and bulk Set-field/Delete in the list header.
Group-by / search / list response Reused logic from today's /media/[typeId] (group_by_value, name fallback) — plus the display-field/title fix.
Type rail New — collapsible navigator listing types via the existing apiListMediaTypes; collapse state persisted in settings.
3-pane shell + record list column New — the explorer layout; replaces the Sidebar + DataGrid + floating EditorPanel composition on the records route only.
Routing Changed — /media becomes the explorer; the active type/record can live in the URL (/media/notes/<id>) for deep links. Files routes untouched.

Carried-over fix

The list still needs a real title — today it shows the raw id (55555555) when there's no name field. The explorer's row title uses the same display-field resolution, so this is fixed as part of the build.

Open questions for the implementation plan

  • Rail content: just record types, or also a top "⌂ Files" entry that jumps back to the Files app? (Leaning: keep it pure-records; Home button returns to the dashboard.)
  • Type icons: auto/none, or a per-type icon picker in type settings? (Mockups assume optional icons; a generic glyph is the default.)
  • Save model: keep today's explicit Save/Delete, or move to autosave like the Files editor? (Mockup shows autosave + "✓ saved".)
  • Deep-link routing: adopt /media/[typeId]/[recordId] now, or keep in-page selection and defer URLs (Backlog Item 20)?
  • Mobile: on narrow widths, collapse to one pane at a time (rail → list → detail as a drill-down)?

If this captures it, I'll write the concrete implementation plan against these answers. Tell me where you land on the open questions (or just say "your call" on any).
