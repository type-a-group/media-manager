Records — Title & Subtitle

Fix the broken “title by” setting and extend it into a configurable title + subtitle per record type.

1 · Why “title by” doesn’t work today

Root cause. The list-row title resolver (recordListTitle()) returns item.name first and only falls back to the chosen field. Any type that has a name field therefore always shows name — the persisted “title by” choice is computed by the server but discarded by the client.

Secondary gaps: the only secondary line a row can show today is the group-by value (ephemeral, and only when grouping is on). There is no way to choose a stable subtitle.

2 · The model

Every record type gets two persisted display settings (in its settings.json):

  • required Title field — the bold primary line. If unset, defaults to the first sensible field (name if present, else the first text field, else the first field). So rows are never a raw id.
  • optional Subtitle field — a muted secondary line. Unset = no subtitle (single-line rows, exactly like now).

Stored as displayField (already exists) + a new subtitleField. Both resolved server-side so the list API returns ready-to-render title_value / subtitle_value; the client stops second-guessing with name.

5 · Behavior rules

Situation Title Subtitle
Nothing configured first field (name → first text → first) — none —
Title set, subtitle unset chosen field — none —
Both set chosen field chosen field
Chosen field empty on a record falls back to first field / id slice hidden for that row
Grouping is on Group headers stay independent (group-by is still an ephemeral list control). The subtitle field is what shows under each row.

Field values render the same way they do for group-by (urls → display name, lists → joined, etc.). A row keeps its missing-file warning.

6 · Scope of the change

  • Storage: add subtitleField to MediaTypeSettingsFile (mirror of displayField).
  • Server: listRecords resolves an effective title (with default) + subtitle and always returns title_value / subtitle_value.
  • Core: simplify recordListTitle to trust title_value first (kills the bug); add recordListSubtitle.
  • API: settings GET/PATCH + the list ?subtitleField= param; client wrappers.
  • UI: the chosen option above in EntitySettingsDialog General tab; RecordListColumn renders the subtitle line. “Title by” leaves the list header entirely (already done).
  • Tests + fixtures + docs: settingsFile round-trip for subtitleField, fixture notes/settings.json, FEATURES.md.
Decisions for you:
  • Settings layout: Option A (two dropdowns) or Option B (grouped + preview)?
  • Does the subtitle replacing the group-value-as-subtitle behavior sound right, or do you want grouping to still override the subtitle when active?
