Make the Records UI match the Media (Files) UI

Grid stays open · clicking a record opens a right-hand side panel · maximize shared components

1 · The goal

Today the Files hub (/files) shows a grid and, when you click a tile, a side panel slides in on the right while the grid stays visible. The Records views (/media/[typeId]) do not do this: clicking a record replaces the grid with a full-width editor. We want records to behave exactly like files — grid + side panel — and in doing so collapse the two divergent implementations onto shared components so future sessions don't re-fork them.

2 · Where the two screens stand today

Files hub — target pattern

┌ Sidebar ─┬──── main ─────┬ panel ┐ │ class │ DataGrid │ File │ │ filters │ (shared) │ Editor│ │ + search │ [tiles…] │ Panel │ │ │ │ (open │ │ │ stays open ◄──┤ on │ │ │ │ click)│ └──────────┴───────────────┴───────┘
  • Self-contained page; owns its own Sidebar.
  • Uses the shared DataGrid.
  • FileEditorPanel.svelte = right <aside> (420px), prev/next chevrons, autosave-on-advance, close.
  • Editor open-state is local (editorFileId), not the global store.

Records views — to change

┌ AppSidebar ─┬──────── main ────────┐ │ record-name │ EITHER JsonRecordGrid│ │ list │ OR full-width editor │ │ + filters │ OR "select a record" │ │ + grid btn │ (they swap, never │ │ + search │ coexist) │ └─────────────┴───────────────────────┘
  • Shell (AppSidebar) lives in +layout.svelte.
  • JsonRecordGrid.svelte is a hand-rolled tile grid — duplicates DataGrid.
  • Editor is the full-width branch of JsonEditorPane.svelte — replaces the grid.
  • Drives everything through the global selection store (gridViewActive, selectedImageId, …).

3 · Proposed target for /media/[typeId]

┌ Sidebar ─┬──────── main ────────┬ panel ┐ │ filters │ DataGrid (shared) │ Record│ │ + search │ [record tiles…] │ Editor│ │ │ │ Panel │ │ │ stays open ◄──────────┤ (open │ │ │ │ on │ │ │ │ click)│ └──────────┴────────────────────────┴───────┘
  • DataGrid replaces JsonRecordGrid. Records map to the side-agnostic GridItem (no thumbnail → file-icon fallback; chips can show group value / key fields; warning = missing-file ref). Deletes ~290 lines of duplicate grid + group/section logic.
  • New RecordEditorPanel.svelte = right side panel mirroring FileEditorPanel's shell: prev/next chevrons + ←/→ keys, autosave-on-advance, close button, scrollable body of schema-driven FieldInputs, Save + Delete. The grid stays mounted behind it.
  • Grid is always shown — the grid-view toggle and the "select a record" placeholder go away. Clicking a tile opens the panel; it doesn't swap the main area.
  • Bulk actions (current "Select" mode, Set field…, Delete) move into the grid's bulkBar snippet, exactly like the files hub's bulk bar.

4 · Reuse / dead-code reduction (your stated priority)

Component Today Plan

Editor side-panel shell Only FileEditorPanel has the chevron/close/autosave-advance header. Extract a shared EditorPanelShell.svelte (header: prev/next, title slot, close; body slot; ←/→ key + autosave-advance plumbing). Both FileEditorPanel and the new RecordEditorPanel compose it.

5 · Other optimizations worth folding in

  • Collapse the selection store's dead legacy. viewMode: 'linked' | 'unlinked' | 'excluded' and gridViewActive are leftovers from the retired image model. Once records use a side panel, several of these become unused — a clean spot to delete them and the ImageListItem-era naming (selectImage, visibleImageIds) in favor of record-neutral names.
  • Unify how the editor open-state is tracked. Files use a local editorFileId; records use the global store. Pick one model for both (local per-page state reads cleanest and matches files) so the two screens stay structurally identical.
  • Consistent empty/loading states via DataGrid's built-in emptyText rather than bespoke "No records to display." markup.
  • Globals stays special-cased. globals is a single-record json singleton with no grid; it keeps GlobalsEditorPane as-is (called out so we don't accidentally break parity — see CLAUDE.md).
  • Docs in the same change: update docs/FEATURES.md (records now grid+panel; JsonRecordGrid retired) and the test-fixtures note if any on-disk behavior shifts (it shouldn't — this is UI-only).

6 · Rollout (so we can stop at a known-good point)

  1. Extract EditorPanelShell from FileEditorPanel (pure refactor; files behaves identically). Verify files still work.
  2. Build RecordEditorPanel on the shell; render it as a side panel from the records page while keeping the old grid temporarily.
  3. Swap JsonRecordGrid → DataGrid on the records page; move bulk actions into the bulk bar.
  4. Restructure the records sidebar per Q1 (drop record-list + grid toggle, or keep).
  5. Delete dead code (JsonRecordGrid, unused selection fields), update docs, run check/lint/test + a manual serve.

7 · Open questions (let's decide these before I build)

Q1 — The left sidebar's record list. Records currently appear as a clickable name-list in the sidebar and can be shown as a grid. If the grid is always visible (like files), that list is redundant. Drop it so the sidebar is filters+search only (matches /files exactly), or keep it as an alternate navigator?
Q2 — Shared panel shell. OK to extract EditorPanelShell and refactor FileEditorPanel onto it (touches a working screen, but removes the duplication you want gone)? Or build RecordEditorPanel standalone and leave files untouched for now?
Q3 — Tile contents for records. Records have no thumbnail. Show the file-icon fallback + name (like a non-image file), and use chips for the group-by value / a key field? Or something denser (a couple of field values as text)?
Q4 — Scope of this pass. Do the full migration now (DataGrid + shared shell + sidebar cleanup + dead-code removal), or land the minimal behavior change first (grid+panel) and schedule the refactors separately?
