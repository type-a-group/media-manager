Per-Entity Icon Picker

Give every class and record type a distinct Lucide icon so the rails, headers, command palette, and file chips become scannable at a glance. This document walks the access workflow and the surfaces that change — review it before we build.

Decisions locked in

Lucide icons only

A curated subset of the Lucide set the app already uses everywhere. Stored as a stable string id (e.g. "film"), validated against the known set, generic fallback when unset/unknown.

stored: lucide id

Four render surfaces

Rails (expanded + collapsed), list/catalog header + breadcrumbs, the ⌘K command palette, and the per-file class chips in the Files grid.

scannable everywhere

User entities only

Classes & record types get a picker (they own the EntitySettingsDialog General tab). All Files, Globals, and the sub-app icons stay fixed.

no reserved entities

Inline beside the name

An avatar-style icon button sits to the left of the Display-name input in the General tab and opens the picker popover.

one shared dialog

Data model — additive, no migration

One new optional field on each side, sitting beside the existing displayName. Absent ⇒ generic fallback. A shared core/icons.ts defines the curated id set, labels, and a resolveIcon(id) helper used by every render surface. 

// src/lib/core/types.ts — ClassConfigSchema
export const ClassConfigSchema = z.object({
  displayName: z.string().default(''),
  gridGroupByField: z.string().optional(),
  displayField: z.string().optional(),
  icon: z.string().optional()      // ← new: lucide id, validated against ICON_SET
});

// src/lib/storage/settingsFile.ts — MediaTypeSettingsFile
interface MediaTypeSettingsFile {
  displayName?: string;
  displayField?: string;
  subtitleField?: string;
  icon?: string;                   // ← new
}

// src/lib/core/icons.ts (new) — single source of truth
export const ICON_SET = ['image', 'film', 'music', 'file-text', 'tag', …] as const;
export function resolveIcon(id?: string): IconName  // fallback when missing/unknown
Why no migration: both config and settings.json already carry displayName and are read through Zod schemas with defaults — adding an optional key is read-compatible with every existing data root. The ClassSummary and MediaTypeSummary DTOs gain an icon passthrough so the rails get it in their existing list calls (no extra fetch).

Implementation checklist (once approved)

  • new src/lib/core/icons.ts — curated Lucide id set + labels + resolveIcon fallback
  • new src/lib/components/IconPicker.svelte — searchable Lucide grid in a shadcn Popover, reused by both adapters
  • edit core/types.ts + storage/settingsFile.ts — add optional icon (+ ClassSummary/MediaTypeSummary passthrough)
  • edit storage/classRepo.ts · storage/mediaTypes.ts — persist & surface icon in config/settings + summaries
  • edit entity-settings/types.ts + adapters.ts — carry icon through load/save for both adapters
  • edit entity-settings/EntitySettingsDialog.svelte — avatar trigger beside Display name (General tab)
  • edit records/RecordsRail.svelte — replace fixed FileText with the type icon (rows + collapsed)
  • edit routes/files/+page.svelte — class icon in the filter list rows + collapsed glyph; catalog header
  • edit Breadcrumbs.svelte — optional leading icon on a crumb
  • edit CommandPalette.svelte — per class/type icon instead of fixed Tag/FileText
  • edit data-grid/types.ts + DataGrid.svelte — chips carry an optional icon id; Files host maps class→icon
  • edit docs/FEATURES.md + test-fixtures/ — record the new field; seed a couple of classes/types with icons

Verification: npm run check + npm run test, then the test-ui-feature skill to screenshot the rail / header / palette / chips against the fixture.
