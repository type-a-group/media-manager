How sections work

A "section" is just a named bucket of fields with an order. Because Globals has no schema, this rides in a single reserved layout key (same mechanism as today's __field_kinds / __field_meta) — no on-disk format change, no migration.

Creating

  • ＋ New section appends an empty named section (e.g. "Untitled"). It persists even with zero fields because section identity/order lives in the layout key, not inferred from field membership.
  • Rename inline via the section's ⋮ → Rename (or double-click the header). Names are display-only labels; the stable id is generated, so renaming never re-keys fields.

Moving fields between sections

  • Drag a row by its ⠿ grip onto another section — a dashed drop-zone shows where it lands. Updates that field's section + position in the layout.
  • Or use the field's ⋮ → Section ▾ picker (no dragging needed; better for keyboard/touch).
  • New fields land in the section whose ＋ Add field you clicked; the top-level ＋ Add field drops into a default section (first one, or "Ungrouped").

Reordering & deleting

  • Reorder sections by dragging the section header grip, or ⋮ → Move up / down.
  • Reorder fields within a section by dragging rows (manual order replaces today's forced A→Z; an "A→Z" sort is offered in settings for people who prefer it).
  • Delete a section via ⋮ → Delete. Prompt: "Move its fields to Ungrouped" (default, non-destructive) or "Delete the fields too." Never silently drops data.

What the layout key looks like

One reserved record key holds section identity, order, and field membership/order. Field values and their type hints stay exactly where they are today. 

// a new reserved key on the globals record, e.g. __layout (JSON string)
{
  "sections": [
    { "id": "sec_site",  "name": "Site",  "fields": ["site_title", "tagline", "theme", "maintenance_mode"] },
    { "id": "sec_links", "name": "Links", "fields": ["social_links"] },
    { "id": "sec_about", "name": "About", "fields": ["about_photo", "contact_email"] }
  ]
}
Reconcile on load (the app's existing lazy-heal pattern): any data key missing from the layout is appended to the default section; any key in the layout that no longer has data is dropped. So hand-edits to data.json and old globals records (no __layout yet) just work — they render as one default section until you organize them. Section membership/order is the layout key's job; type stays in __field_kinds and options/multiselect/itemType stay in __field_meta — unchanged.

The non-inline type menu

Pulling type out of the row means the everyday view is just key → value. Structure editing (rename, type, type-specific options, section, delete) collapses into one per-field ⋮ popover — the small floating card in the mock. This mirrors how Records splits schema from data, but at per-field granularity and on demand.

  • Cleaner default: no type dropdowns or options inputs cluttering rows you're not restructuring.
  • Add field reuses the same popover — name + type + section in one place, instead of the bottom "Add field" strip.
  • Escape hatch: a settings toggle re-enables small inline type chips next to keys (the "Inline chips" toggle above) for people who restructure constantly.
  • Reuse: the value control is the existing FieldInput.svelte; the options/multiselect/item-type bits can borrow from the schema editor rather than re-rolling them.

What goes in Globals settings (⚙)

Globals is a singleton, so its gear is thinner than a class/type's settings dialog — no rename, no delete, no separate schema tab. It's sub-app prefs, persisted in globals/settings.json (consistent with how each json type and media/ store their settings).

Recommendation & next step

Ship the sectioned property table, Flat by default, types-in-menu by default.

Flat headers keep a small config record calm; the collapsible-cards mode is a one-line setting for when it grows. Type-in-menu makes the 90% case (tweaking a value) frictionless, and the inline-chips toggle covers the restructuring power-user. The only new persistence is the __layout reserved key + a few globals/settings.json prefs — no schema, no migration, autosave policy unchanged.

  • Build order: ① rows → key+value only with a ⋮ field menu; ② sections + __layout read/reconcile; ③ drag-to-move + section CRUD; ④ globals/settings.json prefs + the gear.
  • Touch-points: GlobalsEditorPane.svelte (the rewrite), fieldKeys.ts (new __layout constant), jsonRepo.ts globals read (carry __layout through), and docs/FEATURES.md in the same change.
