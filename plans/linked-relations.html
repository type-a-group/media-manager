Class-filtered file fields & two-way linked relations

Two related additions. (1) A file field can be scoped to a single class — the picker then offers only that class's members (a "class field"). Optional, exactly like recordType on a record field. (2) A record field on a class and a class-scoped file field on that record's type can be linked into one two-way relation: pointing a member file at a record from one side writes the reciprocal reference on the other, so the record automatically lists every member that points at it. Symmetric, writable from both ends.

0 Locked decisions

QuestionDecision
File-field class filterOptional classId on a file field. When set it's a hard limit — only members of that class are selectable; a non-member ref is flagged invalid.
Link sync modelTwo-way writable — both sides editable; an edit on either side writes the reciprocal reference on the other. (Not a read-only derived backlink.)
Source of truthSymmetric — neither side is canonical. The stored state is the union of both sides' assertions; writes keep them equal transactionally.
PairingExplicit, and the counterpart field must pre-exist. You create the class-scoped file field first, then pick it as the inverse on the record field. Linking writes the reciprocal pointer on both schemas.
Single-side conflictReassign (move) — setting a new value on a single-cardinality side steals the link from the previous owner; the old record drops it. No error states.
Scope of v1: a link is exactly one shape — a class C's record field (recordType = T) paired with type T's file field scoped to C (classId = C). No class↔class, no record↔record, no globals links (globals is a schemaless singleton — its file fields may still use the plain classId filter, just not linking).

1 The picture

Worked example used throughout: a class photos and a json type albums.

  class photos (members are blobs)             type albums (json records)
  ┌──────────────────────────────┐         ┌──────────────────────────────┐
  │ field album  : record → albums │  ⇄ link │ field shots : file → photos    │
  │                              │         │   (classId = photos)         │
  └──────────────────────────────┘         └──────────────────────────────┘

  blob abc.album = album "Trip"   ──writes──▶  album "Trip".shots ∋ abc
  album "Trip".shots += def      ──writes──▶  blob def.album = "Trip"
  (album single on photos side ⇒ moving abc to "Beach" drops it from "Trip")

Because both sides are stored (writable, symmetric), each side renders from its own data with the machinery that already exists — the record field resolves via recordRefs.ts, the file field via the manifest. The only genuinely new code is the propagation that keeps the two stores equal on every write, plus the schema-editor wiring.

2 Data model core

All additive in src/lib/core/types.ts (FieldDefinitionSchema already has recordType + multiselect). Backward-compatible — every new prop is optional.

  • classId?: string — for a file field, the class whose members the picker is limited to. Ignored for non-file types. (Feature 1.)
  • linkedField?: string — the counterpart field key on the partner schema. Present on both linked fields and self-describing:
    • on a record field: partner type is recordType, partner field is linkedField;
    • on a file field: partner class is classId, partner field is linkedField.
    Both must be set and point at each other for the pair to be "linked." (Feature 2.)
  • Mirror classId + linkedField into AddFieldRequestSchema and UpdateFieldRequestSchema (lines ~409/446 already carry recordType/multiselect).
  • New read-time annotation for the hard class filter (parity with missing_file_fields): out_of_class_fields?: string[] on JsonListItemSchema/FileItemSchema and _out_of_class?: Record<string,string> (key → "not a member of <Class>") on the record schemas. Distinct from missing (the blob exists; it's just not a member). P2 — see §4.
No on-disk migration. Linked fields store ordinary file/record id references — the same bytes a non-linked field stores. classId/linkedField live in the schema only. An old data root opens unchanged; a field becomes "linked" the moment both pointers are set.

4 Feature 2a — link setup in the schema editor frontend server

The counterpart must pre-exist, so the flow has a fixed order; the editor enforces it.

  Step 1 — on type albums: add file field shots, set Class = photos.
  Step 2 — on class photos: add record field album, recordType = albums.
           ☑ Link to inverse field
              counterpart ▸ [ albums · shots  (file → photos) ▾ ]

Eligibility (what the counterpart dropdown offers)

Editing record field R on class C with recordType = T, the counterpart list = file fields F on type T where F.classId === C and F.linkedField is unset or already === R (can't steal an existing link). If none qualify, show "create a class-scoped file field on T first" with a deep link. The link control only appears on the record side (the side that knows both recordType and the counterpart) — the file side shows a read-only "🔗 Linked with photos · album" badge once paired.

Commit writes both schemas (reciprocal)

Selecting the counterpart is one user action but two schema writes, done server-side in updateSchemaField:

// PATCH /api/classes/photos/schema  field=album { linkedField: "shots" }
set  classes/photos.album.linkedField = "shots"
set  records/albums.shots.linkedField  = "album"   // reciprocal, same request

Unchecking, or deleting either field, clears both pointers (never leave a half-link — see §6). The API handlers (classes/[id]/schema, media-types/[typeId]/schema) pass classId/linkedField through; the reciprocal write is performed in the repo so a single endpoint call keeps both schemas consistent under their locks.

Backfill on link: when a pair is first linked, existing references are likely one-sided (people set album on photos before the link existed). The commit runs one reconcileLink() pass (§5's union rule) so both sides immediately agree. Cheap — one class file + one type's data.json.

5 Feature 2b — the propagation engine server

New module src/lib/storage/relationLinks.ts — the only structurally new logic. It is invoked after any write that can change a linked field's value, and it mirrors the change to the partner store.

// Given an entity whose linked field F changed value old → new,
// compute the edge diff and apply the reciprocal writes on the partner.
propagateLinkedChange({
  side:        'class' | 'type',
  ownerId:     // classId or typeId of the edited entity
  rowId:       // blob id (class side) or record id (type side)
  field, def,  // the linked field + its FieldDefinition (has linkedField, partner)
  before, after // the field's id-array value, normalized
}) {
  added   = after − before
  removed = before − after
  for (t of added)   partner(t).addRef(rowId)      // + cardinality-move on single sides
  for (t of removed) partner(t).removeRef(rowId)
}

Where it hooks in

Originating writePartner write
classRepo per-blob metadata set (album changes on blob abc)add/remove abc in each albums record's shots
jsonRepo.updateRecord (shots changes on album "Trip")set/clear album = "Trip" on each blob's photos row
classRepo.removeMembers / blob deletedstrip the blob from every linked record's shots (§6)
jsonRepo.deleteRecordclear that record id from every linked blob's album (§6)

Three invariants the engine must hold

  • No re-entry (loop guard). The partner write is applied raw — it does not re-trigger propagateLinkedChange. The originating change already represents both halves; re-propagating would loop. Implement as an internal writeRaw path the engine calls, distinct from the public mutators.
  • Cardinality move. When the partner side is single (!multiselect) and already points elsewhere, adding the new ref first removes the old edge on both stores (the "steal"), then adds the new. Diff math handles it: setting abc.album = "Beach" yields removed = {"Trip"}, added = {"Beach"}.
  • Deterministic lock order (no deadlock). A linked write touches a class file and a type's data.json (and the manifest). Acquire locks in a fixed canonical order — manifest first (existing rule), then the remaining files sorted by absolute path — so two concurrent linked writes can never hold-and-wait in opposite orders. Document the order next to withFileLock.

Reconcile rule (drift & backfill)

Symmetric stores can drift from hand-edits or a crashed half-write. reconcileLink(C, R, T, F) defines the canonical state as the union of edges asserted by either side: edge (blob, record) exists iff blob.R contains record or record.F contains blob. It then enforces the union on both sides, and for any single-cardinality side that ends up over-full, keeps one edge deterministically (by last_modified, else lexicographic id) and drops the rest from both stores. Run it on link creation (§4 backfill) and from the repair endpoints.

6 Cascades & schema lifecycle server

The dangerous edges. Each must clear the partner side so a link never points at something gone.

EventRequired cleanup
Blob removed from class (removeMembers) — its album row disappearsstrip that blob from every linked record's shots
Blob deleted from disk (deleteFromDiskById)same, across every linked type
Record deletedclear that record id from every member blob's album
Linked field deleted (either side)clear the counterpart's linkedField (downgrade it to a plain field; existing values kept as inert ids)
Class deletedclear linkedField/classId on any type file field that pointed at it; that field degrades to "Any file"
Type deletedclear linkedField/recordType on the class record field; it degrades to an unconfigured record field (refs read as missing — existing behavior)

typeIds and classIds are immutable, so rename never breaks a link (only displayName changes). The recordType/classId pointers stay valid.

7 Read & render — mostly free server

The payoff of writable-symmetric over a derived backlink: each side reads from its own stored references, so the existing resolvers already render both ends.

  • The class album field → resolved to the album's title by recordRefs.ts (shipped with the record field). Unchanged.
  • The album shots field → resolved to filenames by the manifest, the same path as any file field. Unchanged.
  • No new read derivation, no per-row cross-store fetch on the hot list path. The cross-store work happens only on write (bounded: the edited row's added/removed edges).

9 Build order

  1. Feature 1 — class filter — classId on the field def + DTOs; FilePicker classId prop → apiListFiles({classIds}); schema-editor Class select for file fields. Outcome: a file field can be limited to one class's members. Independently shippable.
  2. Out-of-class validation (P2) — out_of_class_fields/_out_of_class annotation in manifest + editor hint. Catches hand-edited non-member refs.
  3. Link config — linkedField on the def + DTOs; counterpart picker on the record side, eligibility rules, reciprocal schema write + unlink-clears-both. Outcome: a pair can be declared; no propagation yet.
  4. Propagation engine — relationLinks.ts: propagateLinkedChange + reconcileLink, loop guard, cardinality move, canonical lock order. Hook into class metadata writes & jsonRepo.updateRecord. Outcome: editing either side updates the other.
  5. Cascades — §6 cleanup on remove-member / delete-blob / delete-record / delete-field / delete-class / delete-type. Outcome: no dangling half-links.
  6. UI polish + backfill — link badge, move toast, summary lines; reconcileLink backfill on link creation and from repair.
  7. Fixtures · tests · docs — see §10.

Phase 1 ships alone (the class filter). Phases 3–5 are the relation; 4 is the heart and where the test weight goes.

11 Risks & edge cases

Deadlock across stores

Two linked writes touching the same class + type in opposite orders. Mitigated by the canonical lock order (manifest → paths sorted) in §5. The single biggest correctness risk; covered by an explicit ordering test.

Half-written propagation

A crash between the two store writes leaves drift. Bounded and self-healing: reconcileLink (union rule) from repair restores agreement; reads still render each side from its own data meanwhile.

Autosave + propagation

Editors autosave and patch the row in place without refetch. A propagated change lands on the other entity, not the open one — so no clobbering of in-flight keystrokes. The other side refreshes on its next list/open (acceptable; it's not the focused row).

Scope creep into many-to-many UIs

v1 is one link shape (class record ⇄ type file). Class↔class and record↔record links are deferred, not designed-in — keep relationLinks.ts parameterized by (side, partner) so they could be added later without a rewrite.
