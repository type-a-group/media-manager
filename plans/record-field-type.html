media-manager · plan

record field type — reference another type's records

A foreign-key field, mirroring the existing file field. A record field stores the referenced record's id; the target record type is configured in the schema, so the picker only searches that one type — never all records. Single or multiselect (a per-field toggle), the value links through to the referenced record, and dangling references are flagged with the same discipline as broken file references. The multiselect toggle is also added to file fields in the same change.

1 Data model core

All in src/lib/core/types.ts. Additive and backward-compatible.

  • Add 'record' to FieldTypeSchema (and the ListItemType note — record is not a valid list item type; multiselect handles multiplicity).
  • FieldDefinitionSchema gains recordType?: string (the target typeId; required-in-spirit for record fields, optional in the zod shape). multiselect?: boolean already exists — its meaning widens from "dropdown only" to also cover file and record.
  • Mirror recordType into AddFieldRequestSchema and UpdateFieldRequestSchema.
  • Stored value shape: single ⇒ a record-id string; multiselect ⇒ string[]. Both already pass the record catchall union (z.string() / z.array(ListItemValueSchema)), so no on-disk schema migration is needed.
  • New annotation keys (parity with _missing_files / missing_file_fields):
    • _missing_records?: Record<string,string> on JsonRecordSchema / ImageRecordSchema — key → expected title (or '') for each broken ref. Read-time only.
    • missing_record_fields?: string[] on JsonListItemSchema and FileItemSchema.
Why separate keys (not reuse _missing_files): the resolution source differs (global blob manifest vs the target type's data.json), and a record can carry both a broken file ref and a broken record ref at once. Keep them distinct; the FieldInput missing/missingName props are already field-agnostic, so the host just feeds the right map per field.

2 Title & missing-ref resolution server

This is the only genuinely new mechanism. A file ref resolves against one global manifest; a record ref resolves against another type's data file — cross-type and async, so it cannot live inside the pure stringifyFieldValue.

New module — src/lib/storage/recordRefs.ts

// Per-request resolver over the target types a schema references.
resolveRecordRefs(schema, records[]) → {
  // 1. collect distinct recordType targets from `record` fields in schema
  // 2. read each target type's data.json once (batched, deduped)
  // 3. build per-type { id → title } using that type's displayField/name
  //    (reuse projectRecordRow's title logic / recordListTitle)
  available:  Map<typeId, Set<id>>,
  titles:     Map<typeId, Map<id, string>>
}

missingRecordFields(rec, schema, available) → string[]   // parity w/ missingFileFields
missingRecordsMap(rec, schema, resolver)   → Record<key,title>? // parity w/ missingFilesMap
recordRefTitles(rec, schema, resolver)    → Record<key,string>  // key → comma-joined titles

Wire the resolver into the three read paths that already annotate file refs:

PathAdd
jsonRepo.getRecordByIdattach _missing_records (so the editor warns)
jsonRepo.listRecordsmissing_record_fields; feed titles into title-by / group-by / verbose field_values
classRepo.listClassMembers + class record getsame, for record fields on a class schema

Rendering titles in lists/verbose (the pure-function gap)

stringifyFieldValue(schema, key, val) only sees the raw id(s). Add an optional 4th param — a resolver resolveRef?(key, val) => string | undefined — that buildFieldValues, projectRecordRow, and fieldSortValue thread through. When present and the field is record, it returns the title(s); absent ⇒ falls back to the raw id. Pure, optional, backward-compatible (every existing caller passing 3 args is unaffected). The server builds the resolver from resolveRecordRefs().titles; the client's optimistic in-place patch passes one built from the RecordPicker's already-loaded titles (or omits it → id fallback until next list load).

7 file multiselect frontend server

Requested alongside the record field. Same toggle, value becomes string[] of blob ids.

  • FieldInput: when multiselect, render a stack of selected-file rows (each a mini FilePicker result + remove) plus an "Add file…" trigger; value is string[]. Single path unchanged.
  • Server: manifest.missingFileFields / missingFilesMap today check typeof v === 'string' only. Extend to also iterate a string[] value (any id not in available ⇒ the field is flagged; the map joins the missing names).

11 Risks & edge cases

Cross-type read cost

Each list with record fields reads the target type's data.json. Dedup per request and batch by typeId in resolveRecordRefs; targets are small JSON files. No N+1 per row.

Cycles & self-reference

A type referencing itself (or a 2-cycle) is fine — resolution is one level deep (title only, never recursive). The schema editor only blocks the trivial self-target to reduce footguns; cross-type cycles are allowed.

Target type deleted/renamed

If recordType points at a type that no longer exists, the field degrades gracefully: picker shows "type unavailable", all refs read as missing. typeIds are immutable, so rename is safe (only displayName changes).

Optimistic patch vs title

After autosave the host patches the row in place without a refetch. The client resolver (from the picker's loaded titles) keeps the rendered title correct; worst case it shows the id until the next list load — never wrong, just less pretty.
